# Attribution

This project was developed with substantial use of AI tools, primarily Claude Code 
(Anthropic's CLI agent) and Claude via the Anthropic API. This document describes 
what was AI-generated, what I authored or modified, and how responsibility is split.

## Tools Used
- Claude Code (claude-opus-4-7) — initial scaffolding and most file generation
- Claude via Anthropic API (`claude-sonnet-4-6`) — used inside the system itself as the toolsmith that dynamically authors new Python tools at runtime

## Authorship Breakdown

### System architecture and design
Authored by me. The README sections up to and including "Architecture" were written 
by me and used verbatim as the prompt to Claude Code. Specifically, I designed:
- the rerouter pattern
- the backend abstraction with UNKNOWN as a sentinel
- letting the agent decide on the best tool to use in MCP fashion instead of using RAG (reason below)
- the experimental setup / evaluation methodology

### Initial code generation
Claude Code generated the initial implementation of all the files in `src/` based on
my architectural prompt, as well as the files in `tests/`.

### My modifications and additions
After initial generation, I made the following changes myself to fix bugs, 
strengthen the architecture, and improve observability:

- **`local_llm.py`** — The original design was to retrieve tools using RAG, allowing
  the model to retrieve tool usage based on the descriptions generated by Claude.
  Testing various thresholds, I learned that `get_current_weather` had a similarity
  score of 0.39 when retrieving for "What's the weather in Tokyo?", while reducing the threshold
  too low caught random tools in its radar. So I let the model decide
  based on the top three retrieved tools, which has so far given me 100% success rates, though it may
  scale differently.

- **Backend rerouter** — added an `UNKNOWN` sentinel to verify the
  rerouter logic was actually firing and not silently defaulting. This caught
  the bug where the model would directly output that it didn't know the answer,
  or that it couldn't do these kinds of tasks.

- **Logging** — added structured logs in `server.py` to debug implicit behavior in the pipeline
  and trace where queries had gone wrong.

- **`catalog.py`** — Removed the unused `get_best_match` function.
  It was a leftover from the RAG prototype that picked the single best skill via
  cosine-similarity threshold. Since a fixed threshold was hard to calibrate, I switched
  to a hybrid model that retrieves the top-3 best matches and lets phi pick among them.

- **`toolsmith.py`** — Iterated on the prompt so the toolsmith stops generating multi-file
  projects or whole codebases and produces a single Python function per tool. Added explicit
  error handling for impossible requests (the infeasibility sentinel). The final change was a
  validating loop that drives the repair process and ensures the generated tool runs as instructed.
