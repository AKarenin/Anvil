# Attribution

This project was developed with substantial use of AI tools, primarily Claude Code 
(Anthropic's CLI agent) for code generation and Claude via the Anthropic API as a 
runtime component of the system itself. This document describes what was AI-generated, 
what I authored or modified, and how responsibility is split.

## Tools Used
- **Claude Code (claude-opus-4-7)** — initial scaffolding and code generation 
  for all files in `/src` and `/tests` based on my architectural prompt.
- **Claude via Anthropic API (`claude-sonnet-4-6`)** — used inside Anvil itself 
  as the toolsmith that generates new Python tools when the local model and the 
  catalog cannot answer a query.

## Authorship Breakdown

### System architecture and design
Authored by me. The README sections up to and including "Architecture" were 
written by me and used verbatim as the prompt to Claude Code. Specifically, 
I designed:
- The three-stage routing pattern (local model → catalog retrieval → toolsmith).
- The backend abstraction with `UNKNOWN` as an explicit escalation sentinel.
- The shift from threshold-based RAG to top-k retrieval with LLM arbitration 
  (rationale described in modifications below).
- The evaluation approach for the system in evaluating tools generated by toolsmith

### Initial code generation
Claude Code generated the initial implementation of all files in `/src`
based on the architectural prompt above. I then worked through the 
system end-to-end and made the modifications described below.

### My modifications and additions

**Retrieval: threshold → top-k with LLM arbitration (`catalog.py`).** 
The initial generation used a single cosine-similarity threshold to decide 
whether the catalog contained a relevant tool. In testing this turned out to 
be brittle: a clearly-correct match like `get_current_weather` for the query 
"What's the weather in Tokyo?" scored only 0.39 — well below any threshold 
that would also reject unrelated tools. Lowering the threshold pulled in 
irrelevant tools; raising it caused valid matches to fall through to the 
toolsmith and re-generate tools that already existed. I removed the threshold 
check and replaced it with top-3 retrieval followed by LLM arbitration: the 
local model picks the best candidate from the three, or rejects all of them. 
In interactive testing across roughly [N] queries spanning [categories], this 
resolved every case where the threshold approach had previously failed.

**`UNKNOWN` sentinel for routing (`local_llm.py`).** The original router 
inferred "the local model couldn't answer" from missing or empty output. In 
practice Phi-3.5 would emit natural-language refusals like "I don't have 
access to real-time data" or "I cannot do these kinds of tasks," which the 
router happily passed back to the user as a final answer. I added an explicit 
`UNKNOWN` token contract: the local model's system prompt instructs it to 
emit `UNKNOWN` for anything requiring live data or capabilities it lacks, and 
the router treats `UNKNOWN` as the only valid signal to escalate. This made 
the routing logic verifiable and caught the silent-failure mode.

**Toolsmith prompt iteration and validation loop (`toolsmith.py`).** The 
initially generated toolsmith prompt produced multi-file scaffolds, full 
codebases with imports and class hierarchies, and occasionally Python notebooks 
instead of single tools. I iterated the prompt to constrain output to one 
self-contained Python function with a JSON input schema. I also added a 
validation-and-repair loop: the generated tool is parsed and checked against 
the schema before being admitted to the catalog, and if validation fails the 
toolsmith is re-prompted with the specific error rather than silently failing. 
Finally, I added explicit error handling for cases where the toolsmith is 
asked to do something impossible (e.g., requires credentials or hardware 
access), so that it returns a structured rejection rather than emitting a 
function that will fail at runtime.

**Logging (`server.py`).** I added structured logs at the routing decision 
point (which backend was selected and why), the retrieval step (top-k scores 
and the arbitration outcome), the toolsmith call (input query, generated tool 
name, validator result), and the executor (tool invocation and result). The 
failure modes above — silent refusals, threshold misses, multi-file outputs — 
were only visible from these logs, not from the final user-facing output.
