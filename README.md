# Anvil

Local agents are my passion: yet they suck. If I tell my phi 3.5 to tell me the weather today, it can't. A common solution is to hardcode it in. If I create a tell-me weather function via python, make the agent run the script, then my local agent can tell me the weather. But I can't hard code any edge cases. Thus, I have created a two step architecture that solves this exact problem: allow a SOTA cloud based model to dynamically program any tools with descriptions, and then use a RAG-based system to find those exact tools for future usage. Here, the local agent will serve as both baseline and the default agent, while the SOTA cloud based model, Claude, will serve as both a tool creator and tool evaluator.

## What It Does

Anvil routes every query through a three-step decision: first, it asks the local Phi-3.5 model whether it can answer on its own; if not, it embeds the query and searches the tool catalog for a semantically-similar stored tool and executes it locally; and only if neither path works does it fall through to Claude, which writes a new reusable Python tool, validates it, adds it to the catalog, and executes it. Each new query class pays the Claude cost exactly once — every similar query afterward is served locally.

## Novelty

Anvil is different from other similar tool-adaptive architectures in different ways. Toolformer (Schick et al., 2023) teaches a model when to insert calls to a fixed API set. Gorilla (Patil et al., 2023) trains a model to map natural language to existing API calls. Voyager (Wang et al., 2023) is the closest prior work as it builds a skill library of Minecraft-domain Python code generated by GPT-4, retrieved by embedding similarity, and reused across episodes. Anvil applies the Voyager-style skill library pattern outside its original game-agent setting and adds two design choices specific to the local-assistant case: (1) a local-first gate where Phi-3.5 must explicitly emit UNKNOWN before any catalog or cloud call is made, so trivially-answerable queries never leave the device, and (2) a cost-amortization framing where Claude is the tool author rather than the runtime executor, so each new query class incurs Claude cost exactly once.

## Architecture

**Local agent.** A thin wrapper around Ollama running Phi-3.5-mini, a personal favorite of mine. It answers directly when it knows the answer, and is instructed to reply `UNKNOWN` when a query requires live data or capabilities it lacks to ensure that every missing answer is probably routed to the toolsmith. It also does parameter extraction from natural-language queries against a tool's JSON schema.

**Toolsmith.** When the local agent returns `UNKNOWN` and the catalog has no match, Claude (`claude-sonnet-4-6`) receives the query along with a strict schema and a few-shot example, and emits a JSON object containing a tool name, description, input schema, and Python implementation. The generated tool is then run through the validator before being admitted to the catalog.

**Catalog and retrieval.** Each tool's description is embedded with `sentence-transformers/all-MiniLM-L6-v2` at registration time and the vector is stored alongside the tool on disk as JSON. Retrieval embeds the incoming query once and ranks tools by cosine similarity; the phi router then dynamically determine by looking at the tool description whether the top three ranked tools were shown, showing a hybrid-RAG system to ensure smarter selection of models. This slowed the process but increased accuracy from tests.

## Evaluation

A 20-prompt evaluation suite spanning math, factual queries, live-data lookups, paraphrased reuse, infeasibility refusals, and intentionally-difficult limitation probes (email actions, Wikipedia summaries, philosophical questions) drove the system's iteration. The first run was diagnostic: the original similarity-threshold gate rejected every cache attempt — `all-MiniLM-L6-v2` simply doesn't reach 0.7 cosine on short paraphrased queries against single-sentence tool descriptions — and the catalog churned rather than accumulated, giving a 0% cache-hit rate even on direct re-queries. Replacing that gate with phi-as-router over the top-3 candidates lifted cache-hit rate to 35% overall and 100% among queries that reached the router; every successful match scored below 0.6 cosine and three below 0.4, none of which would have routed under the old threshold. A second verified eval (with per-prompt verifier functions: exact-match for math, substring for facts, plausibility ranges for live data) returned 16 PASS / 3 WARN / 1 FAIL, with zero spurious regenerations and worst-case latency dropping from 419 s to 35 s.

A separate adversarial probe surfaced a multi-layer failure: the query *"What is the current KRW price compared to USD today?"* routed to a crypto-price tool (shared "fiat currencies" vocabulary), the parameter extractor invented `coin_id="bitcoin"` to satisfy the schema, and the synthesizer dressed up the result confidently. Two guards now sit on that path. An extraction-grounding check rejects required-parameter values that don't appear in the user's query, catching invented fields like the bitcoin case. A `verify_tool_fit` gate asks phi a yes/no after routing — *"does this tool's described capability actually allow you to answer the task?"* — and abandons the cache hit on NO, catching semantic mismatch the router rationalizes through. Together they cut the wrong-answer rate on the adversarial set from 3/3 to 1/3 while preserving every legitimate cache hit. The residual case is the ceiling of phi3.5's gating capacity: when the query and the tool description share enough vocabulary, a 3.8B model rationalizes the fit. Escalating the verification gate to a larger model would close that gap and is documented as future work.
